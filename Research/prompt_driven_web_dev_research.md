# Wild Animal Music as a prompt-driven Jekyll substrate

The recommendation in one breath: stay on Jekyll, push page copy into `_data/`, install Sveltia CMS at `/admin` with a Personal Access Token plus the Anthropic Claude GitHub Action as Marie's chat fallback, keep CLAUDE.md under 120 lines as a real policy file, and put a Cloudflare Worker behind AI Gateway in front of any future talking-character feature. Every other decision below is in service of those four. The site stays a vanilla Jekyll repo on the default GitHub Pages flow with no CI, every editor verb maps to one or two file edits, and none of the foundation choices preempt the art-direction calls John has not made yet.

## 1. Repo conventions for productive Claude Code work

The canonical Anthropic guidance is at [code.claude.com/docs/en/best-practices](https://code.claude.com/docs/en/best-practices) and [code.claude.com/docs/en/memory](https://code.claude.com/docs/en/memory). The single most quotable rule from the best-practices page: "For each line, ask: 'Would removing this cause Claude to make mistakes?' If not, cut it. Bloated CLAUDE.md files cause Claude to ignore your actual instructions." The empirical instruction-budget research that HumanLayer summarises ([humanlayer.dev/blog/writing-a-good-claude-md](https://www.humanlayer.dev/blog/writing-a-good-claude-md), citing [arxiv.org/pdf/2507.11538](https://arxiv.org/pdf/2507.11538)) gives a hard ceiling: frontier models reliably follow ~150 to 200 instructions, the Claude Code system prompt already eats roughly 50, and once your CLAUDE.md plus skills push past that envelope, instruction-following degrades uniformly across all rules, not just the new ones. **Brevity is a hard constraint, not aesthetics.**

The hierarchy worth knowing: `~/.claude/CLAUDE.md` is user-level across every repo, `./CLAUDE.md` is the project file checked into git, `./CLAUDE.local.md` is gitignored personal notes, subdirectory `CLAUDE.md` files are pulled in on demand, and the new auto-memory system writes a separate `MEMORY.md` ([medium.com/@joe.njenga, Anthropic auto memory](https://medium.com/@joe.njenga/anthropic-just-added-auto-memory-to-claude-code-memory-md-i-tested-it-0ab8422754d2)) that you can disable with `CLAUDE_CODE_DISABLE_AUTO_MEMORY=1`. CLAUDE.md is what you tell Claude, MEMORY.md is what Claude infers. Don't confuse them.

Real-world examples on GitHub converge on a tight shape. The Microsoft `CopilotStudioSamples` Jekyll site runs 204 lines ([github.com/microsoft/CopilotStudioSamples/blob/main/CLAUDE.md](https://github.com/microsoft/CopilotStudioSamples/blob/main/CLAUDE.md)) and only earns it because it documents a multi-pattern content taxonomy. HumanLayer's own root file is reportedly under 60 lines, and the `abhishekray07/claude-md-templates` repo ([github.com/abhishekray07/claude-md-templates](https://github.com/abhishekray07/claude-md-templates)) treats 80 lines as the ceiling. **For a two-person band site, target 60 to 120 lines.** Repeated sections across the corpus: a one-paragraph "what this is" with the live URL, an annotated 10-to-15-line directory tree, the 2 to 4 build commands, an explicit "how to add a new post / show / release" with the literal front-matter block, named data files called out as canonical sources, a short "do not edit" list, and the git/PR convention.

Anti-patterns are loud and consistent. Don't dump README.md into CLAUDE.md or `@README.md` import it (that re-embeds the README every session). Don't write personality instructions ("be a senior engineer") because the system prompt handles that. Don't encode rules a linter or formatter can enforce, HumanLayer's line is "never send an LLM to do a linter's job." Don't ship the unedited output of `/init`. Don't list every file in `_includes/`. And critically, don't trust `.claudeignore` as a security boundary: issues [#1304](https://github.com/anthropics/claude-code/issues/1304) and [#30810](https://github.com/anthropics/claude-code/issues/30810) confirm it isn't enforced, and [The Register reported in January 2026](https://www.theregister.com/2026/01/28/claude_code_ai_secrets_files/) that Claude Code v2.1.12 reads `.env` files even when both `.gitignore` and `.claudeignore` list them. The hard mechanism is `.claude/settings.json` permission deny rules.

For the Project-as-control-plane / CC-as-executor pattern specifically, the clearest articulation is Zillatron27's [claude-code-control-plane](https://github.com/Zillatron27/claude-code-control-plane), explicitly written by a non-developer running multiple repos: "Claude Code is a stateless execution engine. The web interface is a planning and thinking layer. Code remains the execution layer. The context repo is the shared state that connects them." That maps cleanly onto John's setup. The Anthropic Project holds long-form artifacts (band brand voice, photo licensing notes, approved phrasings); the repo holds CLAUDE.md plus the actual Jekyll site; plain-English asks flow Project to Code as either pasted instructions or, with the GitHub MCP server ([github.com/github/github-mcp-server](https://github.com/github/github-mcp-server)), as direct read/write. Anthropic's internal teams report ([www-cdn.anthropic.com PDF](https://www-cdn.anthropic.com/58284b19e702b49db9302d5b6f135ad8871e7658.pdf)) makes the cadence explicit: "have claude write commits as it goes for each task step. This way either claude or I can revert to a previous state if something goes wrong." For a content site driven by a non-technical editor, the right grain is one commit per logical content change.

Skip `.claude/skills/` at this scale. Anthropic's own framing ([anthropic.com/engineering/equipping-agents-for-the-real-world-with-agent-skills](https://www.anthropic.com/engineering/equipping-agents-for-the-real-world-with-agent-skills)) is that skills are for "specialized domain knowledge that Claude needs sometimes but not always." At wildanimalmusic.com volume (a few posts a month, occasional shows), there is no "sometimes but not always" workflow. A single CLAUDE.md plus 1 to 3 slash commands in `.claude/commands/` (e.g. `/new-post`, `/add-show`, `/swap-hero-image`) is the right shape. Do add a `PostToolUse` hook in `.claude/settings.json` that runs `bundle exec jekyll build --safe` after edits to `_posts/`, `_pages/`, `_data/`, `_layouts/`, `_includes/`, or `_config.yml`. Per the docs, "hooks are deterministic and guarantee the action happens," and that catches Liquid syntax errors before Marie sees a green PR.

Concrete recommendations for this repo: target a 60-to-120-line CLAUDE.md, include an explicit "How requests arrive" section that names the operator/executor split so Claude treats spec drafts from the Project as source of truth, encode commit-per-logical-change as a rule, and add the build hook. Put brand voice in a separate `BRAND.md` referenced by pointer ("for tone questions, read BRAND.md first") rather than inlined, so it loads on demand rather than every session.

## 2. Non-coder editing workflows that actually survive Marie running them solo

Decap CMS (formerly Netlify CMS) is alive but slow. Latest is v3.11.0 from March 24, 2026 ([decapcms.org/docs/releases](https://decapcms.org/docs/releases/)), repo is at [decaporg/decap-cms](https://github.com/decaporg/decap-cms), maintained by "PM TechHub & friends" volunteers. The official Jekyll starter has not been touched since August 2024. It works with plain Jekyll, but the GitHub Pages problem is real: Netlify Identity is gone, so Decap's GitHub backend needs an OAuth proxy. Options are a free Cloudflare Worker proxy ([sterlingwes/decap-proxy](https://github.com/sterlingwes/decap-proxy)) or keeping a Netlify account purely as the OAuth provider. The UX is desktop-first React, ~1.5 MB, functional but dated, and the mobile experience is rough.

Sveltia CMS is the project actually moving. v0.128.5 in production beta, v1.0 GA targeted mid-2026, single maintainer Kohei Yoshino ([github.com/sveltia/sveltia-cms](https://github.com/sveltia/sveltia-cms), [sveltiacms.app/en/docs/intro](https://sveltiacms.app/en/docs/intro)). It's a Svelte rewrite of Decap: ~300 KB versus Decap's 1.5 MB, GraphQL to GitHub for fast loads, genuinely usable mobile UX, drop-in compatible with Decap's `config.yml` (you swap one `<script src>` line). **Critically, it supports a Personal Access Token sign-in flow, paste a token once, no OAuth proxy required** ([sveltiacms.app/en/docs/backends/github](https://sveltiacms.app/en/docs/backends/github)). The bus factor of one is the real risk; the mitigation is that the config format is Decap-compatible, so if Sveltia stalls, swap one script tag back. Chris Ayers' [mobile CMS write-up](https://chris-ayers.com/posts/mobile-cms-on-github-pages/) walks through exactly this pattern on GitHub Pages.

Tina CMS is the wrong stack. Tina is React/Next-first, official Jekyll support requires a Node build step plus TinaCloud account with `CLIENT_ID` and `READ_TOKEN` in Actions secrets, and the [scottleechua/tina-ghpages-starter](https://github.com/scottleechua/tina-ghpages-starter) only works on user sites. Pricing in 2026 is a free tier (2 users, 500 assets/month), Team at $29/mo, Business $299–$749/mo ([tina.io/pricing](https://tina.io/pricing)). Skip.

CloudCannon is the polished commercial path. Free developer plan, paid plans $25–$45/site/mo plus $10/user ([cloudcannon.com/pricing](https://cloudcannon.com/pricing/)). The site stays on GitHub Pages: CloudCannon connects to the repo, edits commit back, GH Pages still builds it ([jekyllcodex.org/blog/cloudcannon](https://jekyllcodex.org/blog/cloudcannon/)). The closest thing to a Weebly-style visual editor for Jekyll, with essentially zero ongoing ops. The trade-off is the subscription and a mild lock-in to CloudCannon's editable-region annotations.

Static CMS is dead, archived September 9, 2024 ([github.com/StaticJsCMS/static-cms](https://github.com/StaticJsCMS/static-cms)). Pages CMS ([pagescms.org](https://pagescms.org)) is a small but credible MIT alternative, free, hosted at app.pagescms.org, GitHub OAuth via a GitHub App (no proxy), mobile-optimized. JekyllPad is hosted SaaS with a 5-post/month free cap, vendor risk on a small operator.

GitHub-native paths fail Marie at the schema layer. The web editor on github.com is fine for typo fixes on desktop and useless for adding a new tour-date row in `_data/tour.yml` because YAML indentation errors silently break Jekyll builds. GitHub Mobile [explicitly cannot create new files](https://github.com/orgs/community/discussions/40852), only edit existing ones. github.dev (press `.`) is full VS Code in the browser, desktop-class on a phone, the wrong abstraction for a non-coder. Pure GitHub-native is a typo-fix fallback, not a primary workflow.

The Claude-mediated landscape is where Marie's existing fluency pays off. **Anthropic Cowork** went GA across paid plans in late February 2026 ([claude.com/blog/cowork-research-preview](https://claude.com/blog/cowork-research-preview), [cnbc.com 2026/02/24](https://www.cnbc.com/2026/02/24/anthropic-claude-cowork-office-worker.html)) but is desktop-first with filesystem access. **Claude for Chrome** ([claude.com/claude-for-chrome](https://claude.com/claude-for-chrome)) is a Chrome/Edge extension on all paid plans and Anthropic explicitly calls out built-in GitHub navigation knowledge: it can drive github.com to edit a file and commit. Still beta, desktop only, prompt-injection warnings. The Claude.ai GitHub connector is read-only ([support.claude.com/en/articles/10167454](https://support.claude.com/en/articles/10167454-using-the-github-integration)): it syncs file contents into a Project but cannot open PRs.

The underrated path is the **`anthropics/claude-code-action@v1` GitHub Action** ([github.com/anthropics/claude-code-action](https://github.com/anthropics/claude-code-action), [code.claude.com/docs/en/github-actions](https://code.claude.com/docs/en/github-actions)). John runs `/install-github-app` once, sets `ANTHROPIC_API_KEY` as a repo secret, and from that moment Marie can open a GitHub Issue from her phone titled "Add tour date" with body "@claude add a show on July 12, 2026 at The Echo, Los Angeles, doors 8pm, tickets at..." and Claude opens a PR with the exact `_data/tour.yml` edit. Cost is API tokens, typically under $5/month at this volume per [groundy.com](https://groundy.com/articles/how-to-run-claude-code-as-a-github-actions-agent-for-automated-pr-fixes/). Marie types into a GitHub Issue from her phone; that's it.

Walking the realistic path "add a tour date" through each option exposes the failure modes. Decap on GH Pages: Marie hits `/admin`, OAuth via the Cloudflare Worker, fills the form, commits. Failure: Worker secrets expire or callback URL drifts and Marie can't self-recover. Sveltia: same flow, plus PAT-only login skipping the proxy entirely; failure mode is PAT rotation every ~12 months, which is John's problem. CloudCannon: log in, visual editor, click Publish; failure is a lapsed subscription. The Action: open Issue, write paragraph, John reviews and merges; failure is occasional schema-mismatch PRs that need correcting.

Recommendations ranked. **Primary: Sveltia CMS at `/admin` with PAT login.** Drop-in compatible Decap config, the only Git CMS that's actually pleasant on a phone, no OAuth-proxy fragility. Cost zero. **Backstop: install the Anthropic Claude GitHub App so Marie can `@claude` from any Issue.** Fifteen minutes of John's time, ~$5/mo, lets Marie use her best skill (chatting with Claude) directly against the repo. **Skip the rest** unless Marie reports the form-driven Sveltia UI feels too rigid for her writing flow, in which case CloudCannon at $25–$45/mo is the upgrade. Don't bother with Decap (Sveltia is strictly better here), Tina (wrong stack), or Cowork/Claude for Chrome as Marie's daily driver (desktop-only, beta).

## 3. The architectural move from Jekyll site to LLM-mediated CMS substrate

The destination is "Carrd built on Jekyll, grown incrementally." The substrate is content-as-data, a small fixed component vocabulary, image-key indirection, and a token file. Each plain-English editor verb resolves to a single canonical file.

Jekyll's `_data/` directory accepts `.yml`, `.yaml`, `.json`, `.csv`, `.tsv`, with subfolders adding namespace levels ([jekyllrb.com/docs/datafiles](https://jekyllrb.com/docs/datafiles/)). For a 10-page music site, the right shape is per-page or per-section files, not one fat `site.yml`:

```
_data/
  pages/
    home.yml          # hero, blurb, featured links
    about.yml         # bio paragraphs, member list
    contact.yml       # email, photo key, booking text
  tour.yml            # list of dates
  releases.yml        # albums, links
  press.yml           # quotes
  images.yml          # logical key -> file path + alt text
  nav.yml             # already exists
```

The CloudCannon walk-through ([cloudcannon.com/tutorials/jekyll-tutorial/introduction-to-jekyll-data-files](https://cloudcannon.com/tutorials/jekyll-tutorial/introduction-to-jekyll-data-files/)) and Hui Jing's JekyllConf 2019 talk ([huijing.github.io/slides/67-jekyllconf-2019](http://huijing.github.io/slides/67-jekyllconf-2019/)) both endorse this granularity at production scale. **The cautionary tale is Dan Gop's "Liquid Templated Data in Jekyll"** ([dan.gop/articles/jekyll-liquid-data](https://dan.gop/articles/jekyll-liquid-data/)): he started with `people.yml` and `projects.yml`, needed cross-file references, tried YAML anchors (which only work within a single file), and ended up enabling Liquid processing inside data files. The lesson: as soon as you reach for anchors or includes inside YAML, you've reinvented a database badly. Keep YAML flat. Duplicate strings across files. If a record needs more than 4 fields, promote it from `_data/tour.yml` to a `_tour/` collection, that migration is one PR.

For style tokens, **bare CSS custom properties in a single file are the correct answer at this scale**. Style Dictionary ([styledictionary.com](https://styledictionary.com/info/tokens/)) compiles one source to iOS, Android, and web, which is overkill when you ship to one platform. Open Props ([open-props.style](https://open-props.style/)) is a curated set of `--font-sans`, `--size-fluid-3`, `--ease-elastic-3` you import via CDN, fine as a starting palette to fork rather than as a runtime dependency. A `:root` block of ~30 lines (color, type, spacing, motion) in `assets/css/tokens.css` is the right size. Token discipline against one-off tweaks is enforced through the prompt surface: tell Claude in CLAUDE.md that all colors come from `tokens.css` and no hex outside that file, and it complies.

For the component vocabulary, Jekyll's parameterized includes (`{% include hero.html title="..." image=site.data.images.hero %}`, accessed as `include.title`) plus a section dispatcher pattern is the move. David Egan's "Page Sections in Jekyll" ([dev-notes.eu page-sections write-up](https://dev-notes.eu/2016/08/page-sections-in-jekyll-seperating-content-from-layout/)) shows the canonical loop:

```liquid
{% for section in page.sections %}
  {% case section.type %}
    {% when "hero" %}    {% include sections/hero.html s=section %}
    {% when "gallery" %} {% include sections/gallery.html s=section %}
    {% when "tour" %}    {% include sections/tour.html s=section %}
    {% when "text" %}    {% include sections/text.html s=section %}
    {% else %}           {% include sections/_unknown.html type=section.type %}
  {% endcase %}
{% endfor %}
```

Pages are composed declaratively in `_data/pages/home.yml` as a list of typed sections. **Target 5 to 7 section types maximum**: hero, text, image, gallery, tour, releases, contact. The set grows too large the moment a new section differs from an existing one only in styling; that's a token tweak, not a new component.

Image asset abstraction is where the GitHub Pages constraint bites cleanly. `jekyll-picture-tag` ([github.com/rbuchberger/jekyll_picture_tag](https://github.com/rbuchberger/jekyll_picture_tag)) generates `<picture>`, resizes, converts to webp/avif, reads presets from `_data/picture.yml`. It's beautifully suited to this use case and **explicitly not whitelisted on GitHub Pages**. The plugin's own deployment docs say so: "GitHub Pages only allows a very short whitelist of plugins, which sadly does not include JPT." The whitelist is the github-pages gem allowlist (jekyll-seo-tag, jekyll-sitemap, jekyll-redirect-from, jekyll-feed, jekyll-paginate). The cleanest pattern on a non-CI Jekyll setup is the lowest-tech one, an image-key indirection table:

```yaml
# _data/images.yml
hero_main:
  src: /assets/img/hero-main.jpg
  alt: "Wild Animal performing at the Echo, Los Angeles"
contact_photo:
  src: /assets/img/contact-2024.jpg
  alt: "Marie and bandmate, photo by J. Smith"
```

```liquid
{# _includes/img.html #}
{% assign i = site.data.images[include.key] %}
<img src="{{ i.src | relative_url }}" alt="{{ i.alt }}" loading="lazy">
```

"Change the contact photo" is now a one-field edit. You give up automatic responsive `<picture>` and webp generation, which for a band site with a dozen images is invisible to users. If image perf later matters, add a 30-line GH Actions workflow ([docs.github.com using-custom-workflows-with-github-pages](https://docs.github.com/en/pages/getting-started-with-github-pages/using-custom-workflows-with-github-pages)) and swap the include's body to call jekyll-picture-tag; the call sites don't change.

Convention enforcement at this scale is mostly a prompt-surface concern, not a tooling concern. The two pieces worth building: the `sections/_unknown.html` warning partial (catches typos in `type:` without bricking the GH Pages auto-build), and a JSON Schema check via [check-jsonschema](https://check-jsonschema.readthedocs.io/en/latest/precommit_usage.html) once `_data/` has 5+ files. The redhat-developer YAML language server ([github.com/redhat-developer/yaml-language-server](https://github.com/redhat-developer/yaml-language-server)) reads `# yaml-language-server: $schema=...` modelines for in-editor validation. ajv in Node, custom Ruby validators in `_plugins/`, Bookshop component definitions, all CMS-team tools, all overhead at 10 pages with one editor.

The prompt-surface design target. Each plain-English verb has a canonical file. "Change the contact photo" edits `_data/images.yml`, key `contact_photo.src` and `alt`. "Add a tour date" appends a record to `_data/tour.yml`. "Make the about-page hero bigger on mobile" edits a per-section CSS custom property override in the about layout, not the global tokens. "Rotate the welcome text" adds a `title_variants` array to `_data/pages/home.yml` and the hero include picks one with `{% assign idx = "now" | date: "%j" | modulo: page.title_variants.size %}`. **Document this verb-to-file map at the top of README.md as the prompt-surface contract**; that README is the actual CMS schema, and Claude reads it.

SSG choice as architectural decision. Honest comparison: image processing built-in is the only structural argument for switching, and it's not overwhelming. The `_data/images.yml` indirection captures 90% of the editorial benefit without a build pipeline. Migration cost from the current Jekyll port is zero for staying, a full template rewrite for any of Eleventy, Astro, or Hugo. Marie has no local-dev needs under any of them. Prompt-surface clarity is highest in Jekyll because Liquid + flat YAML is the stack with the smallest gap between editor verb and file edit. Astro's `.astro` files mix JSX, frontmatter, Markdown, and components; that's more surface area for Claude to navigate, not less. Koos Looijesteijn's writeup ([kooslooijesteijn.net/blog/jekyll-or-astro](https://www.kooslooijesteijn.net/blog/jekyll-or-astro-which-is-better-static-site-generator)) lands here: Jekyll is feature-complete for sites with limited technical requirements, Astro shines for component-heavy JS-interactive sites. Wildanimalmusic.com has one toggle of JS. **Verdict: stay on Jekyll.** If perf forces it later, adding `.github/workflows/pages.yml` (~30 lines, [dev.to deploy-jekyll-to-github-pages-2026](https://dev.to/pooyagolchian/deploy-jekyll-to-github-pages-in-2026-github-actions-custom-domain-cloudflare-50fb)) is a smaller commit than rewriting templates.

Migration path, staged. **PR 1: extract page copy to `_data/pages/*.yml`**, layouts now read `{% assign p = site.data.pages[page.name] %}`. Stop. Live with it for two weeks. This single PR unlocks 80% of the LLM-editing value because any text change becomes a YAML edit. **PR 2: image-key indirection**, create `_data/images.yml` and `_includes/img.html`, replace every bare `<img>` in includes. **PR 3: section dispatcher with 3 to 4 section types**, convert one page (home) to a list of sections; don't convert all pages, prove the pattern. **PR 4 (optional, defer until pain): JSON Schema + check-jsonschema in a tiny Action that runs on PRs only**, not for deploy. Where to stop: don't introduce a CI build pipeline just to enable jekyll-picture-tag, don't write Ruby plugins (breaks default GH Pages build), don't migrate every page into `_sections/` collections, don't add Bookshop or Decap until there's a second editor refusing git.

## 4. A no-regrets foundation for the art layer

The current low-JS posture is the right starting grain. As Jeremy Keith puts it, "the web platform does not require build toolchains. Always default to, and regress to, the fundamentals of CSS, HTML, and JavaScript" ([adactio.com](https://adactio.com/)). The cleanest layering primitive in 2026 is HTML web components: wrap working markup in a custom element that adds behaviour, never the reverse. Jim Nielsen's example ([blog.jim-nielsen.com html-web-components](https://blog.jim-nielsen.com/2023/html-web-components/)) wraps a regular `<img>` inside `<user-avatar>` so the page is meaningful without JS, then upgrades when `customElements.define` runs. Naming convention in the wild: prefix the wrapper with the element it enhances (`button-confirm`, `a-geolocate`, `character-walker`). Skip Shadow DOM unless you need slot-based composition; light DOM keeps existing CSS cascading.

A build step buys you something only when you need npm-only ESM with bare specifiers, or once `/assets/js/` exceeds 6 to 8 modules and waterfall requests hurt. Until then, plain `<script type="module" src="/assets/js/character.js">` works on GitHub Pages. Modern animation libraries publish `+esm` URLs: `import { animate } from "https://cdn.jsdelivr.net/npm/motion@latest/+esm"` ([motion.dev/docs/quick-start](https://motion.dev/docs/quick-start)), and Anime.js v4 documents both the ESM and UMD paths ([npmjs.com/package/animejs](https://www.npmjs.com/package/animejs)). When the build day comes, esbuild standalone (single Go binary, no config, run from a Makefile or 10-line GitHub Action) is the smallest viable answer. Vite wants to own the dev server and is the wrong shape for a Jekyll repo.

Library landscape, current as of April 2026. **GSAP went fully free under Webflow on April 30, 2025**, including all formerly paid plugins (SplitText, MorphSVG, ScrollTrigger), under Webflow's Standard License with one prohibition on building competing visual no-code products ([gsap.com/pricing](https://gsap.com/pricing/), [css-tricks.com gsap-now-free](https://css-tricks.com/gsap-is-now-completely-free-even-for-commercial-use/)). Core is roughly 23 to 35 KB minified, CDN-friendly, no build step. Anime.js v4 is MIT, native ESM with subpath imports, slightly smaller than GSAP. **Motion (motion.dev, formerly Motion One) ships a `motion/mini` build at 2.3 KB** using WAAPI only, and a hybrid `animate` at ~18 KB with sequences and an animation loop ([motion.dev/docs/upgrade-guide](https://motion.dev/docs/upgrade-guide)). Howler.js for audio is ~7 KB gzipped, drop-in `<script>`, handles iOS unlock ([howlerjs.com](https://howlerjs.com/)). Pixi.js v8 is WebGPU-first 2D rendering, several hundred KB full bundle, warranted only for sprite-based characters with many frames or shaders. Phaser 4 is 345 KB gzipped full, a game framework, the wrong shape for occasional hidden interactives. Theatre.js authors animations in a browser GUI then ships only `@theatre/core` to production, genuinely good for keyframed character vignettes. Lottie is 60 KB gzipped (`lottie-web@5.13`), shines when an illustrator hands you an After Effects export.

For "1 to 5 small interactives over the next year": **GSAP plus Howler covers 90% of what an indie art site wants**. Reach for Motion mini only if your interactions are purely transform-and-opacity. Defer Pixi, Phaser, Theatre.js, Lottie until a concrete art piece demands them.

LLM backend hosting for Berlin, hobby scale, no first-party "deploy a Claude character" service from Anthropic. Cloudflare Workers free plan is 100,000 requests/day, 10ms CPU, no credit card; paid is $5/month for 10M requests ([developers.cloudflare.com/workers/platform/pricing](https://developers.cloudflare.com/workers/platform/pricing/)). Cold starts are effectively zero in 2026 ([blog.cloudflare.com eliminating-cold-starts-2-shard-and-conquer](https://blog.cloudflare.com/eliminating-cold-starts-2-shard-and-conquer/)). Berlin latency: PoPs in Berlin, Frankfurt, Hamburg, low single-digit ms round trip. Setting the secret is one command: `npx wrangler secret put ANTHROPIC_API_KEY`. Cloudflare AI Gateway gives you free observability, caching, rate limits, and a kill switch by swapping the `baseURL` on the Anthropic SDK ([developers.cloudflare.com/ai-gateway/usage/providers/anthropic](https://developers.cloudflare.com/ai-gateway/usage/providers/anthropic/)). Vercel Hobby tier is free but [explicitly non-commercial only](https://vercel.com/docs/plans/hobby), fuzzy for a band that sells merch, and edge functions billed per ms make a 10-to-30-second LLM stream burn through GB-hours. Netlify moved to credit-based pricing on September 4, 2025 (Free 300 credits/mo, Pro $19/seat with 3,000), runs on AWS Lambda with 200 to 500 ms cold starts, noticeable for a chat backend. **Cloudflare Workers behind AI Gateway is the right answer.**

Failure modes need design, not afterthought. The chat character should always render and animate without ever calling the LLM; the LLM is a progressive enhancement that fails to a hand-written line pool. Per-IP rate limits at the edge using Workers KV (10/min, 100/day starter), per-session token caps (`max_tokens: 256`, refuse inputs over 500 tokens), server-side persona system prompt locking the character's name and rules, no tool use whatsoever. The OWASP LLM Top 10 ([genai.owasp.org/llmrisk/llm01-prompt-injection](https://genai.owasp.org/llmrisk/llm01-prompt-injection/)) puts prompt injection at #1 with no fool-proof prevention; least-privilege removes most of the surface. Use Anthropic prompt caching (`cache_control: { type: "ephemeral" }`) on the persona block for ~10% input pricing on cached portions ([dev.to whoffagents deploy-claude-api-on-cloudflare-workers](https://dev.to/whoffagents/deploy-claude-api-on-cloudflare-workers-edge-native-ai-with-durable-objects-and-kv-18a8)). Pick the cheapest model that still feels like the character; Haiku for casual chat, Sonnet only if the personality lift is worth it. Keep a JSON file of 30 to 50 hand-written lines in the character's voice as the fallback; ship the character with that pool on day one before wiring up the LLM at all.

**The architectural blueprint and staged migration plan**:

- **PR 1**: create `_data/pages/{home,about,contact}.yml`, move every paragraph, heading, and CTA out of HTML into YAML; layouts read via `{% assign p = site.data.pages[page.name] %}`. Stop, live with it two weeks.
- **PR 2**: create `_data/images.yml` plus `_includes/img.html`; replace every bare `<img src="/assets/...">` in includes with `{% include img.html key="..." %}`.
- **PR 3**: section dispatcher in `_layouts/page.html` with `_includes/sections/{hero,text,image,tour}.html` and a visible `_unknown.html` warning partial; convert just `home.yml` to a list of sections to prove the pattern.

- **PR 5** (when image perf becomes measurable, possibly never): add `.github/workflows/pages.yml`, switch deploy to a custom workflow, swap `_includes/img.html` body to wrap `jekyll-picture-tag`. Call sites don't change.

Tokens land in `assets/css/tokens.css` as ~30 lines of `:root` custom properties from PR 1 onward. Bare CSS, no Style Dictionary, no Open Props as a runtime dependency, copy ten props you like into your own file and delete the upstream reference if you want a starting palette.

**A baseline architecture for the art layer** (no-regrets, doesn't preempt later art-direction calls):

- *Progressive enhancement primitive*: HTML web components, registered from a single `/assets/js/components/index.js` loaded as `<script type="module">`. Every interactive starts as working static markup wrapped in a custom element that upgrades on `connectedCallback`.
- *Default toolkit*: GSAP for animation (free since April 2025) plus Howler for audio, both as CDN `<script>` tags, no build step. Reach for Motion mini (2.3 KB) only when your first interactive is purely transform-and-opacity. Pixi, Phaser, Theatre.js, Lottie are earn-it-later.
- *LLM backend* (the day a chatty character ships): Cloudflare Worker behind Cloudflare AI Gateway, `ANTHROPIC_API_KEY` as a secret via `wrangler secret put`, ~50 lines of vanilla JS using the Anthropic TS SDK ([platform.claude.com/docs/en/api/sdks/typescript](https://platform.claude.com/docs/en/api/sdks/typescript)) with `cache_control: ephemeral` on the persona block. Free tier covers hobby scale by 100x, Berlin edge ~5ms, $0 unless usage explodes.
- *Graceful degradation by construction*: the character renders and animates entirely without the LLM; chat is a click-to-enhance affordance that falls back to a 30-to-50-line JSON pool of in-character canned lines on any non-200, on rate-limit, on JS not loaded. Per-IP KV rate limits, `max_tokens: 256`, server-locked system prompt, no tool use. Ship the character with the canned pool first, wire up Claude only when the persona feels right.
- *Defer the build step*: plain ESM modules in `/assets/js/` until waterfall loading hurts; then esbuild standalone from a Makefile or a 10-line Action. Don't introduce Vite for a Jekyll repo.

The shape of the recommended setup is one durable swap from the current state (copy out of HTML, into `_data/`), one drop-in CMS line in `<head>` (Sveltia), one GitHub App install (`@claude`), and a CLAUDE.md that documents the verb-to-file map. Everything else is a deferred decision. Marie can run the site solo from a phone within a week, John keeps the Projects/Code workflow he already uses, and nothing in the foundation closes the door on the talking character or the moving illustrations when the art direction is ready.
